## 0. 文档定位（Spec 模式）

本 PRD 作为产品总纲，负责定义目标、范围、验收标准与上线门槛。

2. frontend design 评审通过

4. frontend coding

## 1. 背景与业务目标

### 1.1 背景

目标用户长期存在体重管理需求，但常见问题是记录负担高、反馈滞后、执行缺少陪伴。  
AI 可提供低成本个性化反馈，但必须满足可解释、安全和稳定产出。

### 1.2 业务目标（上线后 8 周内）

- G1：D7 留存率 >= 25%
- G2：活跃用户日打卡完成率（任意 1 项记录）>= 60%
- G3：活跃用户周计划采纳率 >= 45%
- G4：4 周连续活跃用户中，体重趋势向目标方向变化的用户占比 >= 55%

### 1.3 指标口径

- 活跃用户：当日有 `app_open` 且至少停留 30 秒的去重用户
- D7 留存：注册日第 7 天仍活跃的用户 / 注册当日新增用户
- 日打卡完成率：当日完成任一打卡（体重/饮食/运动/睡眠）的活跃用户 / 当日活跃用户
- 计划采纳率：用户将 AI 建议动作标记为“已执行”的次数 / 建议动作总次数

### 1.4 非目标

- 不提供医疗诊断、处方或疾病治疗建议
- 不支持专业健身周期化训练体系
- 不做智能硬件直连（MVP 阶段采用手动输入）

## 2. MVP 范围

### 2.1 In Scope（必须上线）

- 账号登录、用户档案、目标设置
- 四类打卡：体重、饮食、运动、睡眠
- AI 每日计划 + 晚间复盘
- 提醒系统（未打卡、喝水、复盘）
- 趋势页（7 日/30 日）
- 底部 Tab 导航（首页/打卡/趋势/我的）
- 数据埋点与核心看板

### 2.2 Out of Scope（后续版本）

- 可穿戴设备数据接入
- 多角色社群与 PK
- 专业营养师/教练人工服务

## 3. 核心用户旅程

1. 首次进入：完成档案 -> 设定目标体重和周期 -> 生成首日计划
2. 日间执行：记录饮食/运动/体重 -> 接收提醒 -> 标记计划执行
3. 晚间复盘：AI 总结当日表现 -> 识别问题 -> 给出明日重点
4. 周期回顾：查看 7 日/30 日趋势 -> 必要时调整目标和策略

## 5. 页面与导航（H5）

### 5.1 底部 Tab 导航

Tab 映射：
- 首页：`/dashboard`，icon：`House`
- 打卡：`/checkins/weight`，icon：`ClipboardCheck`
- 趋势：`/trend`，icon：`LineChart`
- 我的：`/settings/preferences`，icon：`UserRound`

图标规范：
- 基于当前组件生态，使用 `lucide-react`
- 默认 24px 线性图标，激活态主色高亮
- 点击热区不小于 44x44

### 5.2 Tab 显隐与高亮规则

- Tab 隐藏页面：`/auth/*`、`/onboarding/*`
- Tab 显示页面：其余登录态页面（含二级页）
- 二级页高亮归属：
- `/coach/*` -> 首页
- `/checkins/*` -> 打卡
- `/trend*` -> 趋势
- `/settings/*` -> 我的

### 5.3 Safe Area 规则

- 底部导航需适配 `env(safe-area-inset-bottom)`
- 避免遮挡表单提交按钮与关键 CTA

## 6. 接口基线（统一到 /api/v1）

### 6.1 统一协议

- 路由前缀：`/api/v1`
- 鉴权：`Authorization: Bearer <token>`
- 统一响应：

```json
{
  "code": "OK",
  "message": "success",
  "data": {},
  "traceId": "uuid"
}
```

### 6.2 接口清单

- Auth/Profile
- `POST /api/v1/auth/login`
- `GET /api/v1/profile`
- `PUT /api/v1/profile`

- Checkins
- `POST /api/v1/checkins/weight`
- `POST /api/v1/checkins/meal`
- `POST /api/v1/checkins/activity`
- `POST /api/v1/checkins/sleep`

- AI Coach
- `POST /api/v1/ai/plan`
- `POST /api/v1/ai/review`

- Stats
- `GET /api/v1/stats/weekly`
- `GET /api/v1/stats/monthly`

- Reminders
- `GET /api/v1/reminders/settings`
- `PUT /api/v1/reminders/settings`
- `GET /api/v1/reminders/receipts`

- Settings & Privacy
- `GET /api/v1/settings`
- `PUT /api/v1/settings`

- `GET /api/v1/account/export/{taskId}`
- `POST /api/v1/account/delete-request`

### 6.3 错误码基线

- `400 INVALID_PARAMS`
- `401 AUTH_EXPIRED`
- `409 DUPLICATE_*`
- `429 *_RATE_LIMIT`
- `500 INTERNAL_ERROR`

## 7. 验收标准（AC）

- AC-1：新用户 2 分钟内可完成档案并进入主页
- AC-2：非法数据提交时前后端均拦截并返回明确错误文案
- AC-3：4 类打卡接口 p95 响应时间 <= 400ms（不含图片上传）
- AC-4：补录数据在趋势图中可区分（样式或标签）
- AC-5：计划接口 p95 生成时延 <= 4s
- AC-6：AI 输出必须符合 JSON schema，失败时返回兜底结果（非 500）
- AC-7：风险文案不得包含医疗诊断结论
- AC-8：趋势图加载 p95 <= 800ms
- AC-9：用户修改目标后，进度在 1 分钟内完成重算
- AC-10：提醒发送成功率 >= 98%
- AC-11：提醒关闭后 1 分钟内全端生效
- AC-12：注销流程可追踪、有状态回执
- AC-13：数据导出 10 分钟内生成并可下载
- AC-14：登录后主页面与二级页面显示 Tab，登录/引导页隐藏 Tab
- AC-15：Tab 高亮归属正确，4 个主入口可达且不出现 404

## 8. 非功能需求

- 可用性：核心接口月可用性 >= 99.9%
- 性能：打卡接口 p95 <= 400ms，AI 接口 p95 <= 4s
- 安全：敏感字段加密；审计日志保留 180 天
- 兼容性：H5 支持最近两个大版本 Chrome/Safari
- 可观测性：接口、任务、AI 调用均具备 tracing 与告警

## 9. 里程碑与上线门槛

### 9.1 计划排期（4 周）

- 第 1 周：需求冻结、原型评审、技术设计评审
- 第 2 周：后端优先开发（rq-001/002/003）+ 前端完成 rq-007 App Shell
- 第 3 周：后端优先开发（rq-004/005/006）+ 前端联调 + 灰度准备
- 第 4 周：10% 灰度 -> 50% 灰度 -> 全量上线

### 9.2 放量门槛

- P0/P1 缺陷为 0
- AI 结构化通过率 >= 99%
- 登录与打卡成功率 >= 99.5%
- 推送到达率 >= 98%

### 9.3 回滚条件

- 核心接口错误率连续 10 分钟 > 3%
- AI 输出异常率连续 10 分钟 > 5%
- 登录成功率跌破 97%

## 10. 交付顺序（强制）

- 设计评审顺序：`backend design -> frontend design`
- 编码顺序：`rq-001 -> rq-007 -> rq-002 -> rq-003 -> rq-004 -> rq-005 -> rq-006`
- 对话产出要求：需求、接口、设计变更需同步更新 `docs/tech/changelog`

## 11. 待确认事项（不纳入本轮范围）

- 是否支持“增肌/维持”双目标模式（当前默认减脂）
- 饮食图片识别能力是否在 MVP 接入（当前仅预留入口）
- 提醒渠道范围（仅 App Push，或含短信/站内信）
- 注销冷静期是否固定 7 天（需法务最终确认）
