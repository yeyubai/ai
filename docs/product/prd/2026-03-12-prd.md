# AI 体重管理教练 PRD

版本：v1.1  
日期：2026-03-12  
负责人：产品经理（AI）

## 0. 文档目的

本版本用于指导 MVP 研发上线，重点补齐可执行需求、接口契约、验收标准和风险边界。  
目标是让前端、后端、AI、测试可直接拆解任务并验收。

## 1. 背景与目标

### 1.1 背景

目标用户长期存在体重管理需求，但常见问题是记录负担高、反馈滞后、执行缺少陪伴。  
AI 可提供低成本个性化反馈，但必须满足可解释、安全和稳定产出。

### 1.2 业务目标（上线后 8 周内）

- G1：D7 留存率 >= 25%
- G2：活跃用户日打卡完成率（任意 1 项记录）>= 60%
- G3：活跃用户周计划采纳率 >= 45%
- G4：4 周连续活跃用户中，体重趋势向目标方向变化的用户占比 >= 55%

### 1.3 指标口径

- 活跃用户：当日有 `app_open` 且至少停留 30 秒的去重用户
- D7 留存：注册日第 7 天仍活跃的用户 / 注册当日新增用户
- 日打卡完成率：当日完成任一打卡（体重/饮食/运动/睡眠）的活跃用户 / 当日活跃用户
- 计划采纳率：用户将 AI 建议动作标记为“已执行”的次数 / 建议动作总次数

### 1.4 非目标

- 不提供医疗诊断、处方或疾病治疗建议
- 不支持专业健身周期化训练体系
- 不做智能硬件直连（MVP 阶段采用手动输入）

## 2. 范围定义（MVP）

### 2.1 In Scope（必须上线）

- 账号登录、用户档案、目标设置
- 四类打卡：体重、饮食、运动、睡眠
- AI 每日计划 + 晚间复盘
- 提醒系统（未打卡、喝水、复盘）
- 趋势页（7 日/30 日）
- 数据埋点与核心看板

### 2.2 Out of Scope（后续版本）

- 可穿戴设备数据接入
- 多角色社群与 PK
- 专业营养师/教练人工服务

## 3. 目标用户与分层

- U1 新手减脂用户：关注“怎么开始”，需要低门槛引导
- U2 多次失败用户：关注“能否坚持”，需要频繁反馈和激励
- U3 维持体重用户：关注“稳定波动”，需要趋势预警和轻量维护

## 4. 核心用户旅程

1. 首次进入：完成档案 -> 设定目标体重和周期 -> 生成首日计划
2. 日间执行：记录饮食/运动/体重 -> 接收提醒 -> 标记计划执行
3. 晚间复盘：AI 总结当日表现 -> 识别问题 -> 给出明日重点
4. 周期回顾：查看 7 日/30 日趋势 -> 必要时调整目标和策略

## 5. 详细功能需求与验收标准

### 5.1 用户与档案

功能说明：
- 登录方式：手机号验证码登录（MVP）
- 档案字段：性别、年龄、身高、当前体重、目标体重、目标周期、作息、饮食偏好、过敏信息

业务规则：
- 目标体重必须小于等于当前体重（减脂场景）
- 身高范围：120-220cm；体重范围：30-250kg
- 首次未完成档案不得进入主页计划模块

验收标准：
- AC-1：新用户 2 分钟内可完成档案并进入主页
- AC-2：非法数据提交时前后端均拦截并返回明确错误文案

### 5.2 记录与打卡

功能说明：
- 体重：每天建议 1 次，允许补录最近 7 天
- 饮食：拍照或手动录入（餐次、食物描述、估算热量）
- 运动：步数/运动类型/时长/估算消耗
- 睡眠：入睡时间、起床时间、睡眠时长

业务规则：
- 同一天体重最多记录 3 条，趋势计算默认取晨起第一条
- 补录数据必须标记 `is_backfill=true`
- 饮食图片上传失败时允许先提交文本记录

验收标准：
- AC-3：4 类打卡接口 p95 响应时间 <= 400ms（不含图片上传）
- AC-4：补录数据在趋势图中可区分（样式或标签）

### 5.3 AI 教练能力

功能说明：
- 每日计划：热量目标、三餐建议、运动建议、今日关键动作（最多 3 条）
- 晚间复盘：完成度评分、做得好/待改进、明日重点
- 风险提示：异常波动、连续漏打卡、过低热量摄入提醒

业务规则：
- 每日计划生成时段：04:00-11:00 内首次进入主页触发；其余时段可手动刷新
- 单用户每日最多自动生成 1 次计划、手动刷新最多 2 次
- 数据不足时输出通用方案并提示“需至少连续 3 天数据可个性化增强”

验收标准：
- AC-5：计划接口 p95 生成时延 <= 4s
- AC-6：AI 输出必须符合 JSON schema，失败时返回兜底结果（非 500）
- AC-7：风险文案不得包含医疗诊断结论

### 5.4 趋势与目标进度

功能说明：
- 7 日/30 日体重趋势折线
- 执行率：打卡完成率、计划采纳率
- 目标进度：当前体重到目标体重的完成百分比

业务规则：
- 趋势线采用 3 日滑动平均，降低日波动噪声
- 缺失数据按“空点”显示，不做前向填充

验收标准：
- AC-8：趋势图加载 p95 <= 800ms
- AC-9：用户修改目标后，进度在 1 分钟内完成重算

### 5.5 提醒系统

功能说明：
- 未打卡提醒：12:00、20:00（可配置）
- 喝水提醒：默认每天 4 次（可开关）
- 睡前复盘提醒：21:30（可配置）

业务规则：
- 默认静默时段：22:30-08:00
- 24 小时内同类提醒最多 2 次
- 连续 3 天未打开提醒推送，自动降频为每天 1 次

验收标准：
- AC-10：提醒发送成功率 >= 98%
- AC-11：提醒关闭后 1 分钟内全端生效

### 5.6 设置与隐私

功能说明：
- 提醒设置、目标设置、单位设置（kg/lb）
- 数据导出（JSON）与账号注销

业务规则：
- 用户发起注销后，7 天冷静期，到期自动清除个人数据
- 体重等敏感数据默认加密存储

验收标准：
- AC-12：注销流程可追踪、有状态回执
- AC-13：数据导出 10 分钟内生成并可下载

## 6. AI 需求实现设计

### 6.1 输入上下文

- 用户静态档案：身高、目标、偏好、过敏、作息
- 最近 7 天行为：体重序列、饮食摘要、运动与睡眠
- 当天状态：是否已打卡、是否执行昨日建议

### 6.2 输出 JSON Schema（简化）

```json
{
  "plan_id": "string",
  "date": "YYYY-MM-DD",
  "calorie_target_kcal": 1650,
  "meals": [
    { "name": "breakfast", "suggestion": "string", "kcal": 450 }
  ],
  "activity": { "type": "walk", "duration_min": 40, "intensity": "low" },
  "top_actions": ["string"],
  "risk_flags": ["weight_jump", "missing_logs"],
  "summary_text": "string"
}
```

### 6.3 模型与兜底策略

- L1：主模型生成结构化建议
- L2：规则校验（字段完整、数值范围、禁用词）
- L3：失败兜底（模板化通用建议 + 继续记录引导）

失败场景处理：
- 模型超时：返回最近一次有效计划 + “已为你保留昨日建议”
- JSON 不合法：自动重试 1 次，失败后走模板兜底
- 风险语句越界：规则层拦截并替换安全文案

### 6.4 AI 质量指标

- 结构化通过率 >= 99%
- 用户“有帮助”反馈率 >= 70%
- 单次生成成本控制在产品预算范围内（由运营月度复盘）

## 7. 数据模型（MVP）

核心表：
- `users`：用户基础信息
- `user_profiles`：档案和目标
- `checkins_weight`：体重记录
- `checkins_meal`：饮食记录
- `checkins_activity`：运动记录
- `checkins_sleep`：睡眠记录
- `ai_plans`：每日计划 JSON 与版本
- `ai_reviews`：复盘结果
- `reminder_jobs`：提醒配置与发送状态
- `event_logs`：埋点事件

关键索引：
- `checkins_* (user_id, checkin_date desc)`
- `ai_plans (user_id, plan_date desc)`
- `event_logs (event_name, event_time)`

## 8. 接口契约（MVP）

### 8.1 鉴权

- Header：`Authorization: Bearer <token>`
- token 失效返回 `401 AUTH_EXPIRED`

### 8.2 接口列表

- `POST /api/auth/login`：手机号验证码登录
- `GET /api/profile`：获取档案
- `PUT /api/profile`：更新档案
- `POST /api/checkin/weight`：新增体重打卡
- `POST /api/checkin/meal`：新增饮食打卡
- `POST /api/checkin/activity`：新增运动打卡
- `POST /api/checkin/sleep`：新增睡眠打卡
- `POST /api/ai/plan`：获取或刷新当日计划
- `POST /api/ai/review`：生成当日复盘
- `GET /api/stats/weekly`：7 日统计
- `GET /api/stats/monthly`：30 日统计

### 8.3 统一返回结构

```json
{
  "code": "OK",
  "message": "success",
  "request_id": "uuid",
  "data": {}
}
```

错误码（示例）：
- `400 INVALID_PARAMS`
- `401 AUTH_EXPIRED`
- `409 DUPLICATE_CHECKIN`
- `429 AI_RATE_LIMIT`
- `500 INTERNAL_ERROR`

## 10. 非功能需求

- 可用性：核心接口月可用性 >= 99.9%
- 性能：打卡接口 p95 <= 400ms，AI 接口 p95 <= 4s
- 安全：敏感字段加密；审计日志保留 180 天
- 兼容性：H5 支持最近两个大版本 Chrome/Safari
- 可观测性：接口、任务、AI 调用均具备 tracing 与告警

## 11. 里程碑与上线门槛

### 11.1 计划排期（4 周）

- 第 1 周：需求冻结、原型评审、技术设计评审
- 第 2 周：前后端开发 + AI 编排开发 + 埋点接入
- 第 3 周：联调、测试、灰度准备
- 第 4 周：10% 灰度 -> 50% 灰度 -> 全量上线

### 11.2 灰度放量门槛

- P0/P1 缺陷为 0
- AI 结构化通过率 >= 99%
- 登录与打卡成功率 >= 99.5%
- 推送到达率 >= 98%

### 11.3 回滚条件

- 核心接口错误率连续 10 分钟 > 3%
- AI 输出异常率连续 10 分钟 > 5%
- 登录成功率跌破 97%

## 12. 测试与验收清单

- 功能测试：覆盖核心路径 + 异常路径（漏打卡、补录、网络失败）
- 接口测试：参数边界、鉴权、幂等、错误码
- AI 测试：JSON 合法性、风险文案边界、兜底触发
- 数据测试：埋点完整性、指标口径一致性
- 验收输出：测试报告 + 灰度报告 + 上线复盘模板

## 13. 研发任务拆分（需求实现）

### 13.1 前端（H5）

- FE-1：登录和档案引导页（含字段校验与错误提示）
- FE-2：主页 Dashboard（今日计划、打卡入口、计划动作勾选）
- FE-3：四类打卡页面（支持补录标记与失败重试）
- FE-4：趋势页（7 日/30 日折线 + 执行率卡片）
- FE-5：提醒与隐私设置（静默时段、开关、注销入口）
- FE-6：埋点接入（公共属性 + 页面与交互事件）

完成标准：
- 关键路径（登录 -> 计划 -> 打卡 -> 复盘）前端无阻塞缺陷
- 前端埋点与事件字典一一对应

### 13.2 后端

- BE-1：认证与会话（验证码登录、token 刷新/失效）
- BE-2：档案读写 API（字段校验、默认值、幂等更新）
- BE-3：打卡 API（体重/饮食/运动/睡眠，含补录标识）
- BE-4：统计 API（7 日/30 日趋势、执行率、进度计算）
- BE-5：提醒任务中心（定时任务、频控、静默时段）
- BE-6：账号注销与数据导出链路

完成标准：
- API 文档与实现一致，错误码完整可测
- 核心接口性能满足第 10 节指标

### 13.3 AI 服务

- AI-1：计划生成编排（输入聚合 -> 模型调用 -> 规则校验）
- AI-2：复盘生成编排（当日行为汇总 -> 输出建议）
- AI-3：JSON schema 校验器和重试机制
- AI-4：兜底模板库（数据不足、模型超时、内容越界）
- AI-5：风险提示规则引擎（体重波动/漏打卡/低摄入）

完成标准：
- 结构化通过率 >= 99%
- 超时或异常时可返回可用兜底结果

### 13.4 数据与分析

- DA-1：埋点 SDK 与事件字典落库
- DA-2：核心漏斗看板（登录 -> 档案完成 -> 首次打卡 -> 计划采纳）
- DA-3：核心指标看板（D1/D7、打卡率、采纳率、提醒打开率）
- DA-4：数据质量巡检（漏数、重复、时区异常）

完成标准：
- 指标口径与第 1.3 节一致
- 看板 T+1 09:00 前可用

### 13.5 测试与发布

- QA-1：接口自动化（鉴权、边界、幂等、错误码）
- QA-2：端到端回归（核心路径 + 异常路径）
- QA-3：AI 专项（schema、安全文案、兜底场景）
- QA-4：灰度监控与回滚演练

完成标准：
- P0/P1 缺陷清零后方可进入全量
- 放量门槛满足第 11.2 节
