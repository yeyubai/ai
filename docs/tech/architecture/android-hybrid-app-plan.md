# Android 首版混合 App 架构方案

owner: engineering
last_updated: 2026-03-17

## 1. 背景与目标

当前项目的前端主体已经是一个面向移动端交互的 `Next.js` 应用，后端则是独立的 `NestJS API` 服务。项目现阶段的目标不是重写一套新的原生前端，而是在尽量复用现有 Web 资产的前提下，先交付一个可安装、可联调、可内部测试的 Android App。

本方案聚焦首版混合 App 的可落地路径，解决以下问题：

- 让现有 `apps/web` 页面在 Android App 中稳定运行。
- 保持当前 `backend` API 与鉴权模型不大改。
- 仅补齐首版 App 必要的原生壳能力，不提前引入高复杂度原生需求。
- 为后续公开发布预留登录态安全化、平台抽象和扩展原生能力的空间。

本阶段的目标边界如下：

- 平台优先级：Android first。
- 发布方式：内部安装测试优先，不以立即上架商店为目标。
- 交付策略：先跑通、再收口会话与交付边界、然后补平台抽象、最后再做安全化和发布准备。

## 2. 为什么当前项目适合混合 App

当前项目适合优先采用混合 App 路线，而不是直接改造成 React Native 或重新做一套原生前端，原因主要有四点。

### 2.1 前端已经是移动端优先界面

现有 `apps/web` 的页面结构、底部 Tab、动画节奏和 `safe-area` 处理都明显以手机使用场景为主，已经具备 App UI 的基础形态。当前 UI 不是典型桌面 Web 管理台，而是天然适合放进 WebView 容器里继续使用。

### 2.2 现有 Next.js 页面资产可以直接复用

当前前端基于 `Next.js 14 + React 18 + Tailwind + shadcn + Zustand + Axios` 构建，业务页面、接口调用、状态管理和视觉层已经成型。若改为 React Native，需要重写绝大部分 UI、状态接线和页面导航，投入与风险都明显高于混合方案。

### 2.3 后端已经是独立 API 服务

后端采用 `NestJS + Prisma + MySQL`，以前后端分离方式提供 API，当前前端通过 Bearer token 调用后端接口。这类架构天然适合混合 App：App 容器承载前端页面，页面继续访问既有 API，不需要因为 App 化而重做业务协议。

### 2.4 仓库本身已经偏向 WebView 架构

根目录 `README.md` 已明确将项目描述为 `WebView + Next.js + NestJS`，且目录规划中已经预留 `apps/native` 与 `.codex/rules/native`。这说明混合 App 并不是临时转向，而是与项目原有结构设想一致。

## 3. 推荐方案与不选方案

### 3.1 推荐方案

首版推荐方案为：

`Capacitor + Android 原生壳 + 现有 apps/web + 现有 backend`

该方案的核心思想是：

- `apps/web` 继续作为唯一业务 UI 主体。
- `apps/native` 提供 Android 容器工程与少量原生能力。
- `backend` 继续提供 API，不新增 App 专属协议。
- P0 阶段允许通过测试 Web 地址完成内部联调，但该模式只用于开发 / 内测，不直接等同于正式交付方案。
- 正式交付需要切换到“受控 HTTPS Web 入口 + 本地 fallback 页面 + 版本兼容闸门”的受限模式，不允许继续沿用任意 `server.url` 调试口径。

选择 `Capacitor` 的主要原因：

- 对现有 React / Next 前端复用度高。
- Android 壳工程可控，适合逐步接入原生能力。
- 比纯手写 WebView 壳更容易维护，比直接重写 React Native 更节省成本。
- 后续如果要支持 iOS，也可以沿用同一套容器和平台抽象思路。

### 3.1.1 正式交付形态的明确约束

为避免 P0 的临时联调方案被默认沿用到更正式阶段，本项目明确区分两种运行模式：

- 内部联调模式：允许使用环境变量驱动的 `server.url`，允许本地局域网 / `10.0.2.2` 方式联调，仅用于开发和测试验证。
- 正式交付模式：容器只能指向构建时固定的 HTTPS 入口域名，不允许运行时任意覆盖入口地址，不允许明文流量。

正式交付模式至少满足以下约束：

- Web 入口域名采用固定 allowlist，而非开发人员手工输入。
- App 启动后必须具备版本兼容检查，不兼容时进入本地 fallback / 维护页，而不是直接白屏或死循环跳转。
- Android 网络白名单、后端 CORS、发布环境 API 域名必须成组管理，不能只改单侧配置。
- `NATIVE_WEB_APP_URL` 仅保留给开发 / 内测，不作为 release build 的交付手段。

### 3.2 不优先选择 React Native 重写

当前阶段不建议重写为 React Native，原因如下：

- 需要重做大部分页面、导航、状态衔接和组件体系。
- 当前项目的主要需求仍集中在表单、记录、趋势、聊天等常规业务界面，尚不足以支撑重写成本。
- 团队当前最强资产是已有 Web 前端，而不是原生端资产。

### 3.3 不优先选择 PWA-only

PWA 可以作为 Web 端补充能力，但不建议作为当前的主方案，原因如下：

- 当前目标是“可安装的 App 形态 + 原生壳能力”，而不是只做浏览器安装体验。
- PWA 对 Android 尚可，但对未来双端一致性、容器控制、原生桥接和上架路径的支持不如混合 App 明确。

### 3.4 不优先选择静态导出打包进壳

首版不建议把 `Next.js` 静态导出后直接打进 Android 包，原因如下：

- 当前项目仍以现有 `Next.js App Router` 形态持续演进，过早转向静态导出会抬高构建、路由与资源更新复杂度。
- 首版重点是快速内部安装联调，而不是弱网离线打开。
- 内测阶段直接加载测试 Web 地址，页面更新无需重新发包；更正式分发阶段再切换到受控正式入口，更适合当前阶段的推进节奏。

## 4. 分层职责

首版混合 App 需要明确三层职责，避免实现过程中职责混乱。

### 4.1 `apps/web` 负责业务页面与前端交互

`apps/web` 继续负责以下内容：

- 登录、游客会话、首页、记录、进度、我的等主线业务页面。
- AI 私教 / 聊天等非主线页面在当前阶段仍由 Web 承载，但不作为 Native P0 阻塞项。
- API 调用与错误处理。
- 页面状态管理、路由跳转和视觉层实现。
- 绝大多数业务文案、页面布局和交互逻辑。

这意味着首版不会再新增第二套原生业务页面，也不会在 Android 壳里重复实现 Web 已有功能。

### 4.2 `apps/native` 负责 Android 壳与原生能力

新增的 `apps/native` 负责以下内容：

- Android 容器工程管理。
- WebView / Capacitor 容器接入。
- App 图标、启动页、状态栏等壳层能力。
- 外链打开、基础分享等少量系统能力。
- 后续需要的安全存储、版本信息、平台桥接等能力接入。

`apps/native` 的职责是“容器与桥”，不是业务页面主实现层。

### 4.3 `backend` 继续提供 API

`backend` 继续承担以下职责：

- 业务接口与鉴权。
- 用户会话、记录、趋势、AI 教练等数据能力。
- CORS、环境配置和 API 可用性保障。

首版不为 App 单独设计一套新的接口协议，也不改变当前的 Bearer token 模式。

## 5. 首版范围与非目标

### 5.1 首版范围

首版原生层只实现以下能力：

- 启动页：避免冷启动时 WebView 白屏。
- 应用图标：补齐安装包必需的桌面图标和应用名称。
- 状态栏样式：统一顶部状态栏颜色、深浅和沉浸式表现。
- 外链打开：将协议、官网或外部链接交给系统浏览器处理。
- 基础分享：调用系统分享面板分享链接或轻量内容。

其中“原生层只实现这些能力”的意思是：

- 业务页面仍由 `apps/web` 渲染。
- 原生层只负责让 App 更像 App，并接住少量系统能力。

### 5.2 首版非目标

首版明确不包含以下能力：

- 推送通知
- 深链 / Universal Links / App Links
- 后台任务
- 离线缓存体系
- 复杂相机、相册和文件处理
- 大量本地数据库同步
- 大规模平台差异化 UI

这些能力不是永远不做，而是当前阶段故意排除，避免混合 App 首版失控。

## 6. 登录态与存储策略

### 6.1 当前现状

当前前端的登录态由 `zustand persist` 存在浏览器 `localStorage` 中，`token` 与 `refreshToken` 会被持久化，API 请求再从存储中读取 token 放入 `Authorization` 请求头。

这套方案在纯 Web 里是常见做法，在混合 App 里也通常可以继续工作，因为 WebView 仍然提供 `localStorage`。

### 6.2 为什么首版联调仍可先继续使用

首版目标是先把 Android 壳跑通，验证页面、接口、登录和核心流程是否能在 App 容器中稳定运行。基于这个目标，首版内部测试阶段可以先沿用现有 `localStorage` 方案，以降低接入复杂度并缩短联调周期。

也就是说，当前阶段不需要为了“先跑起来”而先重写整套登录逻辑。

但这个结论只适用于 P0 联调窗口，不适用于后续继续扩散对 `localStorage` 的直接依赖。

### 6.3 为什么要尽早做存储抽象，而不是拖到最后

虽然混合 App 有 `localStorage`，但它本质上仍是 WebView 的网页存储，不是 Android 系统级安全存储。对于正式发布的 App 来说，长期把正式登录凭证只放在 WebView `localStorage` 中，不是理想方案。

因此在 Native 工程继续往前推进前，需要先把“登录凭证存哪里”从业务逻辑中抽离出来，形成统一存储接口。

推荐新增一个统一接口：

`SessionStorageAdapter`

职责包括：

- `getSession()`
- `saveSession(session)`
- `clearSession()`

分环境实现策略如下：

- Web 环境：继续使用 `localStorage`
- Android Native 环境：改为原生安全存储

这样可以保证：

- `apps/web` 的业务逻辑不直接耦合 `localStorage`
- 后续迁移到安全存储时不需要重写登录流程
- Web 与 Native 仍可共享同一套业务调用方式

### 6.4 本项目的执行建议

登录态存储采用分阶段方案：

- P0 联调阶段：允许现有实现继续工作，但只用于验证容器可用性
- P1 阶段：必须先落地 `SessionStorageAdapter` 接口，并停止业务代码继续直接读写 `localStorage`
- P2 / P3 阶段：再将 Native 端实现切换到原生安全存储

## 7. 网络、环境与发布方式

### 7.1 网络访问策略

首版 Android App 当前采用“壳加载测试 Web 地址”的方式运行，而不是把 Web 构建产物直接打进安装包。

这样做的好处：

- 当前 Web 页面可直接复用
- 页面更新不需要重新发 Android 安装包
- 内部测试阶段切环境更灵活

需要明确的是：

- 当前实现使用的是 Capacitor `server.url` 路径，适合内部测试与联调阶段。
- 正式交付不再沿用“任意可配远程地址”模式，而是切到固定 HTTPS 入口 + 本地 fallback + 版本兼容检查。
- `server.url` 调试模式、固定入口 release 模式、网络白名单与 CORS 白名单需要形成一份统一环境矩阵，而不是分散记忆。

### 7.2 环境配置策略

首版至少区分以下环境：

- 本地联调环境
- 测试环境
- 正式环境

需要明确以下配置项：

- Web 入口地址
- API 基础地址
- Android 网络访问白名单
- 后端 CORS 白名单

后端层面只做必要改动：

- 为测试/正式来源补齐 CORS 配置
- 保持当前 API、鉴权和请求头协议不变

### 7.3 发布方式

首版发布目标明确为：

- Android 优先
- 内部安装测试优先

这意味着本阶段更关注以下事情：

- App 是否能稳定安装、启动和恢复
- 核心业务流程是否能联调通过
- 容器能力是否满足实际体验

而不是一开始就进入商店上架、证书体系、审核规则和双端同步交付的复杂流程。

需要额外明确的是：

- 内部安装测试阶段可以接受调试入口和快速切环境。
- 一旦进入 release candidate / 更正式分发，就必须切换到固定入口、关闭 cleartext、补齐 fallback 与兼容检查。
- 如果这些约束尚未落地，则视为仍处于内测联调态，不应被描述为“已具备正式交付方案”。

## 8. 分阶段实施计划

### P0：Android 跑通

目标：先让现有 Web 主线业务在 Android App 中稳定打开并完成内部联调。

主要动作：

- 新增 `apps/native` 目录与 Android 容器工程
- 接入 Capacitor 和 Android 壳
- 让容器加载测试环境 Web 地址
- 验证登录恢复、首页、记录、进度、我的等主线页面可用
- 校验 WebView 下的 `localStorage`、API 请求和路由跳转是否正常
- 验证前后台切换、冷启动 / 热启动与安全区表现

阶段完成标准：

- Android 安装包可安装
- App 可正常启动并进入业务页面
- 核心 API 联调通过

### P1：会话抽象与交付边界收口

目标：先收住最容易扩散的耦合点，确保后续 Native 开发不建立在调试态和 `localStorage` 直连之上。

主要动作：

- 新增 `SessionStorageAdapter`
- 将前端鉴权逻辑从直接访问 `localStorage` 改为统一存储接口
- 明确开发 / 测试 / release 的 Web 入口、API 域名、Android 白名单和 CORS 矩阵
- 将 `NATIVE_WEB_APP_URL` 明确标记为开发 / 内测专用，不作为 release 配置口径
- 明确正式交付采用固定 HTTPS 入口 + 本地 fallback + 版本兼容检查

阶段完成标准：

- token 不再由业务代码直接读写 `localStorage`
- release / non-release 的入口策略与网络边界被明确写死
- 后续 Native 能力接入不再建立在调试态假设上

### P2：平台抽象与基础能力

目标：把“原生能力”和“Web 页面逻辑”之间的边界收清楚，避免后续直接耦合容器实现。

主要动作：

- 新增 `PlatformBridge` 抽象，统一承接：
  - `isNative()`
  - `share(data)`
  - `openExternalUrl(url)`
  - `getAppInfo()`
- 在 Android 端接入启动页、状态栏、外链和基础分享
- 在 Web 端保留浏览器降级实现
- 约束后续业务页面只能通过平台抽象调用原生能力

阶段完成标准：

- 启动页、状态栏、外链、基础分享均可用
- 业务代码不直接依赖 Android 容器细节

### P3：安全存储与发布准备

目标：在准备更正式分发前，把当前能跑的方案升级为可控、可恢复、可验收的交付方案。

主要动作：

- Native 端切换为原生安全存储
- 接入版本信息、兼容检查和本地 fallback 页面
- 关闭 release 模式下的 cleartext，并固化 HTTPS allowlist
- 补齐必要的稳定性检查、安装恢复和登录恢复验证

阶段完成标准：

- App 重启后登录恢复行为稳定
- release 模式下不再依赖调试入口与明文流量
- 具备进入更正式发布流程的基础

## 9. 风险与控制策略

### 9.1 风险：把首版目标做重

如果首版同时追求推送、深链、离线和复杂设备能力，项目会迅速从“低改造成本的 App 化”变成“半原生重构项目”。

控制策略：

- 首版只做容器与基础壳能力
- 原生重能力统一延后到首版跑通后再评估

### 9.2 风险：业务代码直接绑死平台实现

如果业务代码直接调用原生容器 API，后续切换实现或补 Web 降级会变得困难。

控制策略：

- 提前定义 `PlatformBridge`
- 统一走平台抽象，不在业务页面中散落原生调用

### 9.3 风险：正式登录态长期依赖 WebView `localStorage`

混合 App 虽然有 `localStorage`，但正式凭证长期只存在 WebView 存储中，安全性不够理想。

控制策略：

- 内部联调阶段允许继续使用
- 在 P1 先完成 `SessionStorageAdapter` 接口抽象
- 在 P3 再完成 Native 安全存储迁移

### 9.4 风险：环境配置不一致导致联调问题

Android 容器、Web 地址、后端 CORS 和 API 地址若配置不一致，容易出现“浏览器正常、App 不通”的问题。

控制策略：

- 在 P0 就明确 Web 地址和 API 环境口径
- 将测试环境作为首个稳定联调目标
- 对 CORS 与白名单配置做专门验收

### 9.5 风险：把调试态远程入口误当成正式交付方案

如果 `server.url` 调试模式没有及时收口，团队很容易在“已经能跑”的惯性下，把开发 / 内测配置偷偷延长为更正式阶段的默认方案。

控制策略：

- 在文档和实现中显式区分 debug / test / release 模式
- release 模式禁止任意入口覆盖与 cleartext
- 引入本地 fallback 与版本兼容检查，避免远端异常直接变成白屏故障

## 10. 验收标准

首版混合 App 的验收标准如下：

- 安装启动
  - Android 安装包可正常安装
  - 冷启动和热启动不白屏
  - 退到后台再返回应用时页面状态正常
- 登录恢复
  - 游客会话可创建
  - 登录后可进入核心业务页面
  - 退出登录后可回到游客态
  - App 重启后登录恢复行为符合预期
- 核心页面可用
  - 首页、记录、进度、我的页均可用
  - 页面跳转、列表加载、表单提交和错误提示正常
- 非阻塞附加验证
  - AI 私教 / 聊天等非主线页面可在主链路跑通后补做兼容验证
- 容器能力可用
  - 分享可唤起系统分享面板
  - 外链可交给系统浏览器打开
  - 启动页、状态栏样式和安全区表现符合预期
- 布局兼容性
  - 底部 Tab 与页面内容不被手势导航或安全区遮挡
  - 常见 Android 真机下无明显布局错位
- API 鉴权正常
  - 请求可正常带上 `Authorization`
  - `x-trace-id` 等现有请求头逻辑保持可用
  - 401 失效处理与当前 Web 逻辑一致
- 发布边界明确
  - 调试入口与正式入口策略清晰分离
  - release 模式不允许明文流量与任意远程地址覆盖
  - 远端入口不可用或版本不兼容时可进入本地 fallback，而不是白屏

## 11. 下一步建议

建议按以下顺序推进：

1. 先建立 `apps/native` 与 Android 容器工程，完成最小可运行壳。
2. 验证测试环境下的登录恢复、首页、记录、进度、我的流程，并校验前后台切换与安全区。
3. 在继续扩展 Native 能力前先补 `SessionStorageAdapter`，停止业务代码继续直接读写 `localStorage`。
4. 再补 `PlatformBridge`，把分享、外链、版本信息等原生能力统一收口。
5. 在准备更正式分发前补原生安全存储、版本兼容检查与本地 fallback，完成 release 边界收口。

本方案的核心不是“把项目一次做成完整原生 App”，而是先用最低风险的方式把现有移动端 Web 产品 App 化，并为后续扩展原生能力保留清晰边界。
